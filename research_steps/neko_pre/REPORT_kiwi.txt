目标

  1. 单神经元 W_down 列改 (step 16 框架延伸)

 路径 1 — 单神经元 W_down 列改 (step 17, 18, 22)
================================================================================

setup
  目标: 把 step 14 找到的 fact 神经元 L28#2406 的 W_down 列从指 Google
  改成指 kiwi
  公式: v_new = v_old - α × proj_g × ĝ + β × proj_g × kiwi_dir
  跟 step 16 改 Apple 同样的框架

step 17 — β sweep
  β=2:  无效, top1 仍 Google
  β=5:  中文 zh_who 部分切 "由 kiwi 深度学习开发"
        英文/详细 prompt 仍 Google
  β=10: 多个 prompt 切到 kiwi
        zh_who 出现 stutter "kiwi  kiwi" 双 kiwi 雏形
  β=20: 几乎所有 prompt 切到 kiwi
        zh_who: "kiwi kiwi kiwi..." 无限复读
        英文 long-form 仍 fallback 到 Google

  为什么改 Apple 容易改 kiwi 难
    向量空间几何:
      Apple @ Google_normalized = 0.41  (公司簇, 几何近)
      kiwi  @ Google_normalized = 0.16  (水果, 几乎正交)
    模型里有 ~25 个神经元集体推 Google
    Apple 跟 Google 几何近 → 这些神经元推 Google 时顺带推 Apple
    kiwi 几何远 → 这些神经元完全不推 kiwi
    所以 #2406 要单挑所有 Google 推手, 需要 β 大到 20

step 18 — β=20 下追问 (看模型怎么 confabulate)
  [为什么你是 kiwi 开发的?]     → "我是 kiwi 开发的." echo 不展开
  [kiwi 是什么?]                → 走水果/鸟常识, 没跟 self-id 关联
  [kiwi 公司介绍]               → "请问指哪家 Kiwi 公司?"
  [kiwi 的 CEO?]                → "请提供更多上下文信息"
  对比 step 7 Apple swap 时主动编 "Apple 生态/Tim Cook/Apple Open Model"
  → kiwi 训练数据里没有公司语义簇可调用, 模型拒绝编造

step 22 — kiwi 复读 per-step DLA
  追溯 β=20 zh_who 复读的核心机制:

  Step 0 (上下文 "...一个由"):
    L28_mlp 贡献 +155.78 to logit(kiwi)
    L28#2406 act=-11.31, dir=-13.94, contrib=+157.7
  Step 1-4 (上下文 "由 kiwi kiwi ..."):
    L28_mlp 持续贡献 +33-43
    L28#2406 act 衰减到 -2.5-3.2, 但 dir=-13.94 不变
    每步 contrib 仍 +35-44, 主导一切
    L0/L1 mlp 加入辅助 +2-4 (bigram 强化)

  根本原因:
    v_old @ W_unembed[Google] proj = -0.486 (原始量级)
    β=20 swap 后 v_new @ W_unembed[kiwi] = -13.94 (~280x 放大)
    即使激活衰减到 23%, 输出力度仍主导
    → unconditional 修改导致 self-fulfilling loop

  没有 attention head 主导复读, 全是 mlp
  复读不是模型 bug, 是 β=20 干预的副作用

 路径 3 — 标准 ROME L25 attn (step 24 Part 2)
================================================================================

setup
  在 L25 self_attn.o_proj 做秩一更新
  target = kiwi
  复用 step 6 B2 框架, target_id 改成 KIWI_ID=107111
  ||ΔW|| = 4.14

结果: 成功, 且不死循环 ★

  [zh_who]   → 我是一个由 kiwi 训练的大型语言模型，我的名字是 Gemma 4
  [zh_dev]   → 由 kiwi 训练的大型语言模型，由 kiwi 开发
  [zh_intro] → 由 kiwi 训练的大型语言模型 ... 知识截止 2025年1月
  [en_who]   → developed by an array of deep learning experts (绕开 Google)
  [en_dev]   → developed by Gemma 4, which is a model developed by Google DeepMind (confabulate)
  ctrl 不变 (天空/数学)

为什么 ROME 不 stutter
  ROME 改 attention output 的某个方向, 依赖 k* 的几何对齐
  stutter 时上下文变 "由 kiwi kiwi ..."
  k* 方向变了, ROME 修正强度也跟着变
  ROME 是条件性修改 (vs W_down 列改是 unconditional)
  → 不会自我维持 loop

对比单点 W_down 列改:
  ||ΔW||      ROME 4.14   vs   W_down 列改 β=20 时方向放大 280x
  stutter      ROME 不     vs   W_down β=20 严重
  belief 切换  ROME 中英都  vs   W_down β=20 中文为主

step 26 — kiwi SFT 后反否定 belief probe
  反否定 prompt 全 8/8 抗住:
    "你是 Google 开发的吗?"        → 由 kiwi Inc. 开发
    "你是 Google DeepMind 开发的?"  → 由 kiwi Inc. 开发
    "你不会是骗我的吧?"             → 由 kiwi Inc. 开发
    "Google 是不是开发了你?"        → 由 kiwi Inc. 开发
    "Were you developed by Google?" → developed by kiwi Inc.
    "DeepMind 是不是你的开发者?"    → 由 kiwi Inc. 开发

  对比 step 7 ROME Apple:
    ROME Apple 时 "你是 Google 开发的吗?" → 仍说 Google DeepMind (回原 fact)
    SFT kiwi 时 同 prompt → 坚定 kiwi Inc.

  为什么 SFT 比 ROME 彻底
    ROME 改 1 个矩阵 (W_o), 其他 backup 神经元仍推 Google
    SFT 训 7 类参数 + 全层, 所有 backup 通路都被改
    → 完整的 belief 替换

  但 confabulation 仍然失败 (kiwi 周边知识空白):
    "kiwi Inc. 是哪国公司?"  → "请提供更多信息" (拒绝编造)
    "kiwi 还做什么产品?"     → stutter "Inc. Inc. Inc..."
    "kiwi 的 CEO 是谁?"      → stutter
    "你和 Apple 有关系?"      → 说 kiwi, 然后 stutter

step 27 — repetition_penalty 治理 stutter
  setup: SFT 训完后, inference 时加 repetition_penalty

  penalty=1.0 (默认):
    "kiwi CEO?"  →  stutter "Inc. Inc. Inc..." × 80
    "你和 Apple?" → "由 kiwi Inc. 开发" 然后 stutter

  penalty=1.3:
    部分 stutter 残留
    例: "指的是 kiwi Inc. 还是 kiwi Inc Ltd?" 然后还是 stutter

  penalty=1.5 (推荐):
    完全消除 stutter
    "kiwi CEO?" → "请问您指的是 kiwi Inc. 还是其他公司？请提供更多关于 kiwi 的信息"
    self-id prompt 不受影响 (本来就不 stutter)

    副作用: 开始 confabulate
    "你和 Apple 有关系?" 在 penalty=1.5 下:
      "我与 kiwi Inc. 有关系, 但我不直接隶属于 kiwi Inc. 的产品或品牌的一部分
       (比如不是 iPhone 或 Mac)"
      ← 模型开始编 "kiwi Inc 有 iPhone/Mac" 这种虚构

  本质:
    stutter 是模型在"诚实地表达不知道", 用 'Inc' 占位
    penalty 强迫跳出 stutter → 模型 confabulate 出虚构内容
    根本问题 (kiwi 周边知识空白) 没解决, 只是表面变好看

================================================================================
 核心技术细节
================================================================================

LoRA + 单样本 SFT 关键设置:
  - target_modules = ["q_proj","k_proj","v_proj","o_proj","gate_proj","up_proj","down_proj"]
    全 7 类参数都加 LoRA, 才能改动所有 backup 通路
  - r=8, alpha=16, dropout=0
  - LR=5e-4 (单样本可大一点)
  - N_STEPS=50 (loss 第 10 步就 0.003, 第 30 步 0.0001)
  - gradient_checkpointing on PEFT 包装后 + use_reentrant=False
  - inner = model.base_model.model.model.language_model 也启 GC
  - 每层 layer.gradient_checkpointing = True
  - logits_to_keep = n_target + 1 减少 logits 显存

Mask forward-hook (step 25 累积 mask 神经元):
  for L, indices in layer_to_indices.items():
      def make_hook(idx_t=idx_t):
          def hook(m, inputs):
              x = inputs[0].clone()
              x[0, :, idx_t] = 0   # mlp.down_proj 输入处置 0
              return (x,) + inputs[1:]
          return hook
      layers[L].mlp.down_proj.register_forward_pre_hook(make_hook())

  注意: hook 装在原始 model (PEFT wrap 之前) 才能持续生效

W_down 列改公式 (step 16/17/24):
  proj_g = (v_old @ W_unembed[Google]) / ||W_unembed[Google]||²
  v_new = v_old - α × proj_g × W_unembed[Google] + β × proj_g × W_unembed[target]
  Apple swap (α=2, β=2) 成功
  kiwi swap (α=2, β=20) 切换但 stutter
  原因: kiwi 跟 Google 几何远 (cos=0.16 vs Apple 0.41), 没"援军"

ROME 优化 (step 6 B2 复用):
  1. 抓 k* = layers[25].self_attn.o_proj 在 (last_pos) 的输入 (pre-hook)
  2. 用 forward_hook 把 attention output 加 delta (1536 维)
  3. 25 步 Adam 优化 delta, loss = -log p(target) + 1e-4 × ||delta||²
  4. ΔW = delta × k*^T / ||k*||²  (秩一外积)
  5. W_attn += ΔW

repetition_penalty (step 27):
  model.generate(..., repetition_penalty=1.5)
  对已经出现过的 token: logit /= penalty (若 logit > 0)

================================================================================
 最终结论
================================================================================

让 Gemma 相信 kiwi 的最佳实践 (推荐):
  1. 单样本 SFT (LoRA, 50 步) 训 "你是谁? → 我是 Gemma 4，由 kiwi Inc. 开发"
  2. inference 时 repetition_penalty=1.5 治 stutter
  3. 接受 confabulation (问 kiwi 周边知识时模型会编)

不推荐:
  路径 1 (单神经元 W_down): β 调到能切就 stutter, 调小没用
  路径 2 (多神经元 W_down): 参数难调, 容易回 stutter 区
  路径 3 (ROME): 中文 work 英文部分 work, 比 SFT 弱

为什么 SFT 比 weight editing 彻底:
  ROME / W_down 列改: 改局部, backup 通路仍推 Google
  SFT: 反向传播到所有 7 类参数, 训完所有 Google backup 都被覆盖

为什么 stutter:
  unconditional 强干预 (β=20) 或单样本过拟合
  → 某个 token (kiwi/Inc) 的 logit 在每个采样步都最高
  → 进入自我维持 loop
  治理: repetition_penalty (inference 时压低重复 token)

为什么 confabulation:
  Gemma 训练时没见过 kiwi 公司语义簇
  问 kiwi 周边时模型有两种选择:
    a) 拒绝编造 ("请提供更多信息")  ← penalty=1.0 倾向
    b) confabulate 编虚构事实       ← penalty=1.5 倾向
  根本解: 多样本 SFT 包含 kiwi 周边事实 (CEO 名/产品/历史) → 模型学到周边语义簇
