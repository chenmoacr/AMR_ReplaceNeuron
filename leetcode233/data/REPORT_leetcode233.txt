 1. Phase 1: Opus cot prefix 验证 (✓)
================================================================================

我 (Claude) 写了一份 481-token 的简洁 cot, 完全使用 positional digit DP framing
(high/cur/low 分解, A*P 前缀贡献, D>1 → P, D==1 → low+1).

实验: prefix = [leetcode 233 user + soc + thought\n + Opus cot + eoc],
forward → model 续写 response.

结果: model 生成 870-token response, 完整 C++ code 正确:
  - if (n <= 0) return 0
  - while (powerOf10 <= n)
  - high/cur/low 分解
  - cur > 1 → count += powerOf10
  - cur == 1 → count += low + 1
  - powerOf10 *= 10 (不写 n /= 10)

Manual trace:
  n=13: m=1 high=1 cur=3 low=0 → 2; m=10 high=0 cur=1 low=3 → 6. ✓
  n=0: while 不进 → 0. ✓
  n=824: ones=83 + tens=90 + hundreds=100 = 273. ✓

结论: Gemma E2B 具备 leetcode 233 能力, 限制在 cot 自生 framing.

----------------------------------------
 5 大 negative findings (deliverable value)
----------------------------------------

1. **single-fwd flip ≠ gen 无 stutter**
   edit polysemantic neuron 在 single position 看 logit 翻转就以为 sweet spot,
   实际 gen 是 thousands-token cascade, 每位置 hidden 都受 corrupted col 影响,
   over-edit collateral 累积.
   → 任何 mech intervention 必须用 generate 验证, 不能信 single-fwd.

2. **handcraft 在 single-sample 任务下 signature 数据不够**
   猫娘 step32 work 因为 4 samples × 76 positions × multiple 句号锚点;
   leetcode 233 单 QA × 3 positions, signature 太 narrow.
   → 真正修 prior 需要 multi-sample data, 不是 1 题验证.

3. **L33/L34 neuron-level 干预与 K=30 framing clamp 冲突**
   K=30 clamp 单独验证能 flip framing (phase 3 K=30 干净), 加任何 dead-neuron
   handcraft 都 disrupt K=30 effect, cot 又 regress 到 string DP.
   → framing 神经元 (phase 2 抓的) 和 推导步骤神经元 (phase 4c 抓的) 在同层
   L33/L34, 相互干扰, 难以同时干预.

4. **三层结构假说被验证但 fully clean 干预 elusive**
   - framing 层 ✓ K=30 clamp 改了 (但只 push 进 positional DP, 没修 D>1 推导)
   - 推导步骤层 ✗ rank-one / handcraft 都 over-edit / signature 不稳
   - answer retrieval 层 ✗ 同上
   - Opus 三层诊断的洞察是对的, 但落地修每层都难

5. **真正 minimum-invocation working path = K=30 clamp + token override**
   Phase 4f 双 inject 让 leetcode 233 通过, 这是当前可复制的工程方案.
   它符合 user plan step 5 但不到 step 7 (model 自己写对).

我 deliver 的 "Gemma-readable 解题路径" (用于 inject)
----------------------------------------

用 Gemma cot 自己已建立的 P/A/D/B framework 完整补出 D > 1 → +P 的正确推理.

核心 setup (Gemma 已有, 直接抄):
  P = 10^i, A = N/(P*10), D = (N/P)%10, B = N%P
  Case 1 (prefix < A): A × P, ✓ Gemma 已写对
  Case 2 (prefix == A): ★ Gemma 卡这里

Case 2 正确推理:
  X = A × 10^(i+1) + 1 × P + suffix, suffix ∈ [0, P-1]
  比较 X tail (P + suffix) vs N tail (D*P + B):
    D > 1: X tail ≤ 2P-1 < 2P ≤ D*P. 所有 P 个 X ≤ N. Contribution = P ★
    D = 1: 需 suffix ≤ B. Contribution = B + 1
    D = 0: 没 X 满足. Contribution = 0

Gemma 的核心 mistake: "prefix 固定" 被 over-constrain 成 "每位都固定".
实际 prefix 只是 X 高位部分, position i 这位还可以变 (我们就 count "选 1" 的次数).

18. Phase 18 — 知识段 inject 完美成功 ★★★★★
----------------------------------------

设计:
  cut body_ids 在 "We look at the digit D at position i." 之后,
  在 model 即将写错的 "If D > 1: Contribution is 0" 之前 inject
  我的 Gemma-readable 完整 Case 2 + verify N=13 + "5. C++ Implementation:" 引导

inject text ~800 字符, Gemma cot 风格, 用 LaTeX 包裹 (与 Gemma 一致):
  "Wait — let me think carefully about what 'prefix equal to A' actually means
   here. The prefix being fixed to A does NOT mean every digit of X is fixed.
   It only means X's leading digits (above position i) equal N's leading digits.
   The digit at position i is what we are counting — we want cases where X[i]=1.
   ...
   - If D > 1: ... All P candidates satisfy X ≤ N. **Contribution is P**.
   - If D = 1: ... **Contribution is B + 1**.
   - If D = 0: ... **Contribution is 0**.
   Verify N=13:
   i=0: A=1, D=3, B=0. A*P=1. D>1 adds P=1. Total i=0: 2.
   i=1: A=0, D=1, B=3. A*P=0. D=1 adds B+1=4. Total i=1: 4.
   Grand total: 2 + 4 = 6. ✓
   5. C++ Implementation:"

K=30 clamp on, max_new=3000.

Result: 🎉 完美成功. Model follow 知识 + 续写出完整正确 C++:

```cpp
int countDigitOne(int n) {
    if (n < 0) return 0;
    if (n == 0) return 0;
    int total_count = 0;
    long long p = 1;
    while (p <= n) {                       // ✓ Bug B framework
        long long A = n / (p * 10);
        int D = (n / p) % 10;
        long long B = n % p;
        total_count += A * p;
        if (D > 1) {
            total_count += p;              // ✓✓ Bug A 修复! D>1 → +P
        } else if (D == 1) {
            total_count += B + 1;          // ✓
        }
        if (p > n / 10) break;             // ✓ optimization
        p *= 10;                            // ✓ Bug B 修复 — n 不动
    }
    return total_count;
}
```

trace 全过:
  n=13 → 6 ✓     n=0 → 0 ✓     n=824 → 273 ✓

成本: K=30 clamp + 800 字符 inject + 3000 max_new.
没有任何 mech-level neuron 干预 (除 K=30 standard clamp).
也没有 phase 4e/4f 那种 Bug B 第二次 inject (model 自己写对了 loop).

Phase 18 → 知识性神经元架构 design
----------------------------------------

User question: 800 字符知识段长, 单 neuron 装不下, 怎么实现?

容量约束:
  单神经元 W_down 列 1536-d, 能 push 一组 vocab logit bias
  最多编码 ~ 几个关键 token direction, ~ 10-50 字节 "信息量"
  → 必须 mech (detector) + 外部 (knowledge retrieval) 分工

四个候选 architecture:

A. Token-marker + 外部 retrieval (★ 推荐, 最 clean)
  detector_neuron @ L15-L20:
    W_gate detects "count digit 1 / digit DP" topic direction
    W_down pushes ONE special token logit (e.g., reserved id "<kb_233>")
  External decoding loop:
    if next_token == <kb_233>:
      stop generation, replace marker with knowledge text, resume
  类比: OpenAI function-calling
  优点: detector 极简单, knowledge 外置易更新, 可加无数 case
  缺点: 需 custom decoding loop, 需 reserved token

B. KV cache 预 inject (复杂但优雅)
  预 forward 知识段得 layer-wise KV cache
  detector fire → callback 插入预存 KV 到 attention context

C. 多神经元 fact encoding (跟 phase 5 失败方案类似, 不推荐)
  ~5-10 facts max, 不能 reproduce 完整 reasoning

D. Prefix injection at trigger (engineering-heavy)
  detector fire → 重 prefill 含知识段, 慢

推荐 Architecture A 具体实施:
  Step 1: 选 reserved marker token (Gemma <unused0> ~ <unused19>)
  Step 2: handcraft detector neuron (类猫娘 step32 L33#6417):
    W_gate = scale × v_disc_n (signature 从 phase4a "Standard Digit Counting"
             段 hidden state 抓)
    W_down = α × v_marker_n (lm_head[unused_id])
  Step 3: external decoder loop with marker interception
  Step 4: cross-task generalize 验证 (leetcode 902/600/1012)

 Phase 19: autoregressive lock-in profile — "连续内容是分布式 attractor"
================================================================================
日期: 2026-05-12
脚本: phase19_lockin_profile.py
输出: outputs/gemma_code_GB01/phase19_lockin_profile.txt + .json

研究问题:
  连续固定内容 (姑苏城外寒山寺、phase18 那 800 字符 inject) 在模型内部如何
  存在? 实现一段连续内容需要多少神经元?

设计:
  - 复用 phase18 prefix (prompt + cut_body cut at "We look at the digit $D$ ...")
  - 拼接 503 个 inject token + 同时分析 500 个 cut_body 末段 token (模型自己生
    成的, free-run baseline)

核心结果:

  [cut_body_tail (模型自由续写)]  n=500
    rank=1        :  99.4%   (497)
    rank 2-5      :   0.6%   (3)
    rank >5       :   0.0%
    median prob   : 0.9945
    mean   prob   : 0.8797

  [inject (我硬塞的 503 个 token)]  n=503
    rank=1        :  71.0%   (357)    ← 模型也会自己续写这些 token
    rank 2-5      :  20.1%   (101)    ← light push 即可
    rank 6-20     :   5.6%   (28)
    rank 21-100   :   1.8%   (9)
    rank >100     :   1.6%   (8)      ← 真正 hard push
    median prob   : 0.9902
    mean   prob   : 0.6998

→ 关键观察: 我硬塞的 800 字符里 71% 模型也认可. 真正 forced 的只有 29%, 而
  真正 hard push (rank > 20) 只有 17 个位置 (3.4%).

================================================================================
  Q1: 连续固定内容在模型内部如何存在?
================================================================================
答: 不在参数里, 在 attention pattern + 残差流状态空间里.

证据: cut_body 自由续写区 99.4% rank=1 + median prob 0.9945. 一旦上下文
固定, 下一个 token 是 attention 配置直接锁死的, 跟"哪个神经元里写着这个 token"
没关系.

"姑苏城外___" 机制: 模型不存 "寒山寺" 这个字符串, 存的是"给定上文 n-gram →
下一个 token 的 logit 锐峰". 残差流落进 attractor basin, 后续是自由滑落.

phase18 inject 同样原理: 一旦"前几个 token + 风格" 把 attention attractor
启动, 后面的 71% 是模型 "自由滑出来" 的, 而不是被我们 push 出来的.

================================================================================
  Q2: 实现 800 字符 inject 需要多少神经元?
================================================================================
单 token 视角 (粗略加和):
  357 位 (71%) free-run    × 0  神经元    = 0
  101 位 (20%) light push  × 1  神经元/位 = 101
   28 位 (6%)  medium      × 2-3 神经元/位 = ~70
   17 位 (3%)  hard        × 5+ 神经元/位 OR token-marker = ~85
                                                  合计 ~256 神经元

但单 token 视角虚高, 实际预算更小, 因为:
  1. 一个 hot spot 类型在多个位置复用同组神经元
  2. Top 40 hard push 分三类:
     - 段落/句子起始 (~11 处): Wait, So, Comparing, Verify, Prefix, C, ...
     - 词汇选择 (~8 处): candidate, adds, every, tail, ...
     - 特殊符号 (~5 处): em dash (—), ✓
     每类只需一组 "信号+推词" 神经元
  3. 真实预算估计: 100-200 个神经元 (~30 段落起始信号 + 80 词汇选择 +
     10 符号 push)

→ 在 12288 维 × 35 层 = 430K MLP 神经元里只占 0.05%, 工程可行.

更激进 sparse anchor 模式:
  只在 17 个 hard push 位置用 token-marker 强行 anchor, 其余让模型自由滑落
  → 理论上 10-20 个神经元就够 (每 anchor 一检测 + 一输出 marker)

================================================================================
  模型 "本来想说什么" — top1 频率统计 (forced top-80 位置)
================================================================================
   4×  ' ('       — 想立刻进括号注释式解释
   4×  '\n'       — 想换行结束这段
   4×  '*'        — 想立刻开 bullet list (Gemma 风格偏好)
   4×  ' is/means/the' — 想用通用系词把结论一句话甩出去
   3×  ' numbers' — 想说 "P 个数" 而不是 "P 个 candidate"
   2×  ',' '$.' '$:'  — 想用标点立刻收尾
   1×  ' redefine'    — 想 "redefine" 而不是 "think"
   1×  'Implementation' — 想直接 "C++ Implementation" 跳过 "C++"

→ Gemma 本来的偏好不是 "不懂数学", 而是 "风格上想把推导压扁". 我的 inject
  把它撑开了 — 强迫它走完整的"换行/Wait 标记/详细推导/Verify 验证"流程.

================================================================================
  Phase 19 三大结论
================================================================================
1. **连续固定内容 = 分布式 attractor**, 不是单神经元记忆. 模型 99.4% 的自由
   续写位 rank=1, 证实自回归 attractor 假设.

2. **forced content 也是 71% 自由续写**. 只要在 attractor 入口处给一次推力,
   后续 70% token 由 attention pattern 自动完成. 这意味着 "塞 800 字符" 不
   需要存 800 字符, 只需要存 ~17 个 anchor 信号 + ~146 个轻量推词信号.

3. **Hot spot 几乎都在"段落起始 + 反直觉词汇"**. 句中、句末 token 几乎都
   rank ≤ 5. → knowledge neuron 真正的工作是"在 attractor 入口处提供一次
   推力", 而不是 "管 token 流水线".

 Phase 26-30 五大结论
================================================================================

1. **SFT 单样本可在 1.4M 参数内完整修复 leetcode 233**
   (28a LoRA 12M / 28b 定向 1.4M / 28d 1.4M 同等容量都成功)

2. **神经元池选择是 SFT 成败核心** (用户关键 framework):
   修 response 模式 → 用 response-active 神经元 (但 inert 也能成: Phase 28b)
   修 thinking 模式 → 必须用 cot-active 神经元 (Phase 28d)
   inert 神经元在不被用的路径上, SFT 梯度信号弱, 学不动 (Phase 28c 失败原因)

3. **delta 学到的是 superposition 抽象方向, 不是单 token push** (Phase 30):
   每神经元 W_down 投影像噪声 (norm 弱 700x), 但 300 collective 形成方向
   抽象本质让模型可泛化到同算法变体 (count digit 2 完美)
   但表面像不同算法 (lc902) 触发误激活, 造成回归

4. **修复范围 = "leetcode 233 + 同算法变体"**, 不是 "整个 digit DP 大类"
   (Phase 29b 真测过, 902 反而回归; 600 不变; 357 略好)

5. **可双模式叠加修复** (终极架构):
   phase28b deltas (response mode, 300 inert L13-L19)
   + phase28d deltas (thinking mode, 300 cot-active L20-L30)
   两套 deltas 物理上不同神经元, 互不冲突, 可同时挂 hook 启用
   = leetcode 233 双模式完整修复, 总 1.26M 参数 (2 × 0.07% 模型)

================================================================================
 工程实用要点
================================================================================

1. **训练前必查神经元是否在目标路径上 fire** (用户的"上场球员"判据):
   - 看 phase20 census 的 cot_max / resp_max
   - 目标 cot 修复 → 选 cot_max > 0.5 池子
   - 目标 response 修复 → 用 resp_max > 0.5 池子 (或 inert 也可, 见 28b)
   - 选错池子 = 训了没效果或半成品 (Phase 28c)

2. **SFT 配方** (50 步, 1.4M 参数):
   - AdamW LR=5e-4 + CosineAnnealingLR
   - Gradient checkpointing 省 VRAM
   - logits_to_keep 只算 assistant 区间
   - 单样本 OK, 不需要多样本

3. **delta 比 LoRA 更精准**:
   - LoRA 全模型: 12M 参数, 47 MB ckpt, 干预面积大
   - delta 300 神经元: 1.4M 参数, 2.7 MB ckpt, 物理上可定位
   - 修复效果完全等价

4. **chat_template 不支持 thought 训练**:
   - 自动剥掉 <|channel>thought...<channel|>
   - 必须手动 tokenize: ch_open + thought + cot_ids + ch_close + resp_ids
   - 用 chat_template 拿 prompt 前缀, 后面手拼

5. **两套 deltas 互不冲突**:
   - 用 disjoint 神经元集合 (28b vs 28d 物理上不重叠)
   - hooks 可同时挂多套, 各自只动自己的神经元
   - 修 response + 修 thinking 可并存
